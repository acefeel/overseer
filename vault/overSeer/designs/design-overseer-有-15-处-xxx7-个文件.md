## 背景
代码中存在大量残留的占位符（如 'xxx'）和临时注释（如 /HACK/BUG/NOTE），这构成了明显的技术债务，严重影响了代码的可读性和可维护性。系统地清理这些标记不仅能提升代码质量，还能迫使团队对已识别但未解决的问题进行正式的跟踪和归档，而不是散落在代码中的碎片信息。

## 建议动作
1. **分析与分类 (Analysis):** 针对所有提及的文件（src/scanners/todo.ts, 以及所有 design-overseer/*.md 文件），使用 grep 或 IDE 的全局搜索功能，精确列出所有 'xxx' 和 /HACK/BUG/NOTE 的具体上下文。
2. **制定处理策略 (Strategy):** 根据标记的性质进行分类：
    a) **已解决/过时:** 直接删除该标记和相关的代码块。
    b) **待办/已知问题:** 不直接删除，而是将其替换为标准的、可追溯的格式（例如，添加一个明确的 JIRA 或 Issue Tracker 链接，并修改注释说明其状态）。
3. **执行清理 (Implementation):** 分文件逐一进行重构和清理。重点关注设计文档中的标记，确保它们的内容是最终版本，而不是临时草稿。
4. **代码审查:** 完成后，必须进行一次全面的 Code Review，确认所有残留的占位符已被处理或替换为正式的 Issue 引用。

## 风险
- 可能误判某些标记的原始意图。有些 'HACK' 注释可能是为了记录临时的、但未来仍需要保留的上下文信息，直接删除可能会丢失关键历史背景。
- 由于文件数量较多且结构复杂（包含代码和多个设计文档），清理过程中存在遗漏或不一致性导致部分标记残留。
