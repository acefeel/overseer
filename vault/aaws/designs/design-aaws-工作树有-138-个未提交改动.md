## 背景
这是一个严重的 Git 工作流卫生问题。工作区存在大量未追踪文件（138个），这表明当前开发状态非常混乱，极大地增加了代码审查和后续协作的难度。必须将这些改动进行结构化处理，通过提交、暂存或忽略的方式清理工作目录，确保项目历史记录的干净和可追溯性。

## 建议动作
执行一次全面的 Git 工作流卫生检查与清理。
1. **分析状态:** 运行 `git status` 和 `git diff --untracked-files`，对这138个文件进行分类。
2. **确定类型:** 判断这些文件是：a) 需要保留的临时资产/日志；b) 应该被忽略的构建产物（如编译目录）；c) 真正需要提交的代码改动。
3. **执行清理策略 (选择其一或组合):**
    * **如果大部分为构建产物:** 将这些文件添加到 `.gitignore`，并运行 `git clean -fdx` 清理工作区。
    * **如果包含关键代码:** 按照功能模块将改动拆分成多个逻辑清晰的提交（Atomic Commits）。
    * **如果是临时实验性修改:** 使用 `git stash push --include-untracked` 进行暂存，等待后续确认后再处理。

## 风险
- 可能误判：如果未正确识别哪些文件是需要保留的资产或配置，可能会导致重要数据丢失。
- 工作流中断风险：在进行大规模清理时，如果未能完全理解所有138个文件的来源和目的，可能会引入不必要的提交噪音。
