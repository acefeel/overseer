## 背景
项目中散布的 TODO 注释代表了分散的技术债务，如果不对其进行集中管理和归档，未来维护人员很容易遗漏或误解这些待办事项。通过系统性地收集、分类和记录这些标记点，可以将其转化为结构化的、可追踪的产品 Backlog，确保技术债得到有效管理。

## 建议动作
1. **全局搜索与收集：** 使用 IDE 或 `grep` 命令在整个项目代码库中执行全量搜索（例如：`grep -r 'TODO:' .`），生成包含所有 TODO 标记的列表。
2. **人工分类与调研：** 对收集到的每个 TODO 实例，分析其上下文和所在文件/模块。确定该待办事项是否已被后续功能覆盖、是否已废弃，或是否仍是当前版本必须处理的关键点。
3. **创建中央 Issue：** 将所有确认的、有价值的技术债务（即需要人工干预的 TODO）汇总到一个专门的 Issue Tracker (如 Jira/GitHub Issues) 中。为每个待办事项分配明确的优先级、所属模块和责任人，并将其标记为“Tech Debt Backlog”。
4. **清理代码：** 在 Issue 创建后，如果某个 TODO 确认已废弃或不适用，则应在对应的文件中移除该注释。

## 风险
- 可能误判：无法确定原始开发者留下的 TODO 的确切业务意图，导致清理时错误地移除了本应保留的临时标记。
- 范围蔓延（Scope Creep）：在调研过程中可能会过度关注低优先级的技术债务，导致任务无限扩大。
